# Premium Real-Time Insights - Финальная версия ✅

## ЧТО ИЗМЕНИЛОСЬ

### ❌ БЫЛО (scheduler подход):
```
Scheduler (9:00, 20:00) → собирает инсайты → сохраняет в профиль
↓ (ждём до следующего сообщения)
Пользователь пишет → AI видит инсайты в промпте
```

**Проблемы:**
- Инсайты могут устареть за 9-11 часов
- Дополнительная сложность (scheduler, background jobs)
- Избыточная нагрузка в фоне
- Инсайты собираются даже если пользователь не общается

### ✅ СТАЛО (real-time подход):
```
Пользователь пишет → собираем инсайты СЕЙЧАС → AI видит в промпте
```

**Преимущества:**
- ✅ Всегда актуальные данные
- ✅ Проще архитектура (нет scheduler)
- ✅ Меньше фоновых процессов
- ✅ Нагрузка только когда нужно
- ✅ Инсайты собираются точно перед использованием

## АРХИТЕКТУРА

### Умное кэширование:

#### 🟢 БЫСТРЫЕ ПРОВЕРКИ (каждый раз, < 0.2 сек):
- **deadline_alert** — близкий дедлайн
- **stuck_task** — задача застопорилась

Выполняются синхронно при каждом сообщении.

#### 🟡 ТЯЖЁЛЫЕ АНАЛИЗЫ (кэш 10 минут):
- **market_opportunity** — тренды в сообществе
- **reverse_matching** — кто нуждается в помощи
- **trend** — долгосрочные тренды

Кэшируются в памяти на 10 минут, пересобираются по истечении.

#### 🔵 СОХРАНЁННЫЕ (из real-time триггера):
- **task_created** — кто-то создал релевантную задачу

Сохраняются в профиль при создании задачи (существующий механизм).

## FLOW

### Когда Premium пользователь отправляет сообщение:

```python
1. autonomous_chat_with_ai() вызывается

2. generate_proactive_context() собирает контекст
   ↓
3. get_premium_recommendations_for_prompt(user_id) — ЗДЕСЬ МАГИЯ
   ↓
4. Проверяем что пользователь Premium
   ↓
5. БЫСТРЫЕ ПРОВЕРКИ (0.1-0.2 сек):
   - Активные задачи юзера
   - Дедлайны через 1-3 дня?
   - Задачи без прогресса 3+ дней?
   ↓
6. ТЯЖЁЛЫЕ АНАЛИЗЫ (с кэшем 10 мин):
   - Проверяем кэш по user_id
   - Если свежий (< 10 мин) — используем
   - Если нет — собираем заново (market, reverse, trends)
   ↓
7. СОХРАНЁННЫЕ (task_created):
   - Загружаем из profile.pending_premium_recommendations
   - Фильтруем shown_count < 3
   ↓
8. Объединяем ВСЕ инсайты
   ↓
9. Группируем по приоритетам (🔴🟡🟢)
   ↓
10. Форматируем для промпта
    ↓
11. AI видит в системном промпте и упоминает естественно
    ↓
12. После ответа: mark_recommendation_shown() для task_created
```

## ПРОИЗВОДИТЕЛЬНОСТЬ

### Быстрые проверки (deadline, stuck):
- **SQL запросы:** 1 (получить активные задачи)
- **Время:** 0.1-0.2 сек
- **Когда:** каждое сообщение

### Тяжёлые анализы (market, reverse, trends):
- **SQL запросы:** 3-5 (анализ всех задач за неделю/2 недели)
- **Время первого вызова:** 0.5-1.0 сек
- **Время из кэша:** 0 сек (мгновенно)
- **Когда:** первый раз → 10 минут из кэша → снова первый раз

### Итого:
- **Первое сообщение после 10 мин:** ~1.2 сек (быстрые + тяжёлые)
- **Все последующие 10 минут:** ~0.2 сек (только быстрые + кэш)

**Это незаметно для пользователя!**

## МИГРАЦИЯ

### Что делать с существующими данными:

**Ничего!** 
- Существующие `pending_premium_recommendations` (task_created) продолжат работать
- Новые инсайты (deadline, stuck, market) будут генерироваться real-time
- Старые scheduler jobs не запуститься (закомментированы)

### Если нужно очистить старые инсайты:

```sql
-- Опционально: очистить старые сохранённые инсайты (кроме task_created)
UPDATE user_profiles 
SET pending_premium_recommendations = NULL 
WHERE pending_premium_recommendations IS NOT NULL;
```

Но это необязательно — старые инсайты просто перестанут показываться.

## FAQ

### Q: А не будет тормозить при каждом сообщении?
**A:** Нет! Быстрые проверки (deadline, stuck) занимают 0.1-0.2 сек. Тяжёлые (market, trends) кэшируются на 10 минут. Итого: первое сообщение ~1.2 сек, остальные ~0.2 сек.

### Q: Что если Premium пользователь не пишет 10 минут?
**A:** Кэш истечёт, но при следующем сообщении обновится за ~1 сек. Это нормально.

### Q: Можно изменить TTL кэша?
**A:** Да! В `premium_simple.py`:
```python
_INSIGHTS_CACHE_TTL_MINUTES = 10  # Измени на нужное значение
```

### Q: А scheduler совсем удалять?
**A:** Можно оставить для ручного запуска (еженедельная аналитика и т.д.). Но для основного flow он больше не нужен.

### Q: Как отключить кэш если нужно всегда свежие данные?
**A:** Поставь `_INSIGHTS_CACHE_TTL_MINUTES = 0` — будет собирать каждый раз.
